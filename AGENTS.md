# AGENTS.md

## Назначение проекта

PoE2 Trade Helper - черновик помощника для торговли в Path of Exile 2.
Цель: искать потенциально выгодные сделки на рынке PoE2, сравнивать цены, находить возможные арбитражные/обменные действия и позже добавить анализ выставленных игроком товаров.

## Рабочая база

- Активный проект - FastAPI-приложение в `app/`: локальный UI в `app/web`, торговый слой в `app/trade2.py` и `app/trade/`.
- `mcp_server.py` - отдельный standalone MCP-сервер со слоем доступа к PoE2 `trade2`: `trade_leagues`, `trade_static_data`, `trade_search`, `trade_fetch`, `trade_exchange`.
- `README.md` хранит краткое состояние проекта для человека; при изменении API-подхода обновляй и README, и этот файл.
- `.env` содержит секреты/локальные значения. Не выводи его содержимое в ответы и не коммить.
- `.env.example` - публичный шаблон переменных.

## Критичные API-факты

- Для обычного поиска сделок через `https://www.pathofexile.com/api/trade2/search`, `trade2/fetch`, `trade2/exchange` отдельный API-ключ обычно не нужен. Нужен корректный `USER_AGENT`.
- Если сеть нестабильна или режет внешние сайты, исходящие HTTP-запросы можно направить через локальный VPN/proxy-клиент: `OUTBOUND_PROXY_URL` задает общий прокси, `POE2_PROXY_URL` имеет приоритет, `OUTBOUND_PROXY_URLS` / `POE2_PROXY_URLS` задают список endpoints для переключения, failover срабатывает на transport-level ошибки и настроенные плохие HTTP-ответы/маркеры недоступности, стандартные `HTTP_PROXY`/`HTTPS_PROXY`/`ALL_PROXY`/`NO_PROXY` остаются рабочими при `OUTBOUND_TRUST_ENV=true`.
- `trade2` - веб-API официального сайта торговли PoE2, но он не является полноценным стабильным официальным developer API. Перед крупными изменениями проверяй живые endpoints.
- Для агрегированных цен PoE2 можно использовать poe.ninja endpoint `https://poe.ninja/poe2/api/economy/exchange/current/overview?league=<league>&type=<type>`. Он сейчас полезнее для Liquid Emotions и Fragment/Boss-entry анализа, потому что возвращает нормализованную цену, объем и 7-дневное изменение.
- В `poe.ninja` exchange overview `sparkline.data` для PoE2 нельзя показывать как абсолютную цену: это процентный ряд. Для графика цены переводить его в положительный ценовой ряд, привязанный к текущей `primaryValue`/нормализованной цене.
- В live UI `get_category_rates` должен сначала пробовать poe.ninja для поддерживаемых категорий, затем откатываться на `trade2/exchange`.
- Официальный API живет на `https://api.pathofexile.com` и требует OAuth Bearer token.
- Для `get_currency_exchange` нужен scope `service:cxapi`; это исторические часовые сводки, не текущий стакан заявок.
- Для персонажей нужен scope `account:characters`.
- Официальные `Account Stashes` и `Public Stashes` в документации помечены как PoE1 only. Для анализа личных ящиков PoE2 не обещай рабочий официальный путь, пока он не проверен заново.
- OAuth-клиенты создаются на `https://www.pathofexile.com/my-account/clients`.
- Для локального десктопного помощника обычно нужен Public Client с Authorization Code + PKCE. Для `service:*` scope нужен Confidential Client и client credentials.

## Подход к разработке

- Предпочитай минимальные, небьющие изменения. Не добавляй абстракции, пока они не убирают реальную сложность.
- Версия приложения хранится в едином источнике `app/version.py` (`APP_VERSION`). При каждом осмысленном изменении кода, UI, поведения API, схемы данных, конфигурации или пользовательской документации обновляй версию в том же изменении по SemVer: patch для совместимых исправлений и внутренних правок, minor для новых совместимых возможностей, major для ломающих изменений или миграций, требующих ручных действий. UI должен брать версию только из этого источника.
- Строго разделяй русскую и английскую локализации UI: весь видимый текст должен идти через `app/web/static/i18n.js` или шаблонные `data-i18n`-ключи. Не смешивай русские fallback-строки в английском интерфейсе и английские fallback-строки в русском интерфейсе.
- Для русских названий валют, фрагментов и других trade2-позиций сначала опирайся на официальный локализованный справочник `https://ru.pathofexile.com/api/trade2/data/static`. Ручные переводы допустимы только как временный fallback, когда официального термина нет.
- В русской версии видимого UI не показывай английские названия предметов, валют и категорий. Технические id можно оставлять только в диагностике или явно технических местах.
- Не строи новую логику на догадках о JSON-структуре. Сначала получи справочник через `trade_static_data` или живой endpoint, затем кодируй по фактической форме данных.
- Сохраняй rate limit дисциплину: явный `USER_AGENT`, короткие батчи, общий limiter по `X-Rate-Limit-*` / `Retry-After`, пауза через `RATE_COOLDOWN_MS`, без агрессивного сканирования рынка.
- Для poe.ninja не запускай Playwright discovery, если известный JSON endpoint отвечает живым запросом; прямой JSON дешевле и надежнее.
- Ошибки внешних HTTP-вызовов в `mcp_server.py` возвращай с понятным `error` и фактическим `status_code`, если он доступен; не логируй тела запросов с потенциально чувствительными данными.
- Для поиска публичных лотов продавца через `trade2/search` используй `trade_filters.account.input` и `sale_type=priced`; `trade2/fetch` батчить максимум по 10 id; в анализ допускай только fetched listings с `listing.price`, `listing.stash` и instant buyout/fixed price типом цены (`~price` или `~b/o`); полный снимок продавца кэшируй по `league/seller/status`, а текстовый фильтр в UI применяй локально к уже полученному снимку.
- Кросс-валютные циклы считай как направленный граф фактических `trade2/exchange` офферов: ребро `A -> B` означает сколько `B` дают за `1 A`; ищи короткие циклы `A -> B -> ... -> A`, применяй зазор/комиссию на каждом шаге, для UI держи `online` как исполнимый режим и не подменяй это сравнением одного предмета в разных валютах оценки.
- Для оценки предметов продавца сравнивай не только название: учитывай тип базы, редкость, item level и официальные stat id из `item.extended.hashes` / `item.extended.mods`; нормализованный текст аффиксов используй только как fallback. Если полностью аналогичных лотов мало, ослабляй сравнение до stat-группы `count` с "аффиксы минус один"; только после этого используй запасной ориентир по типу, уровню и слабому пересечению статов.
- Если пользователь вручную пометил свойства выбранного лота, передавай эти official stat id в оценку похожего рынка: важные свойства должны быть обязательной основой сравнения, лишние исключай из trade2 stat-фильтров и post-filter. Основу предмета учитывай отдельным режимом; для крафтовой оценки поддерживай base-only поиск без свойств, а фокус по tier/level аффикса и диапазону roll-значений применяй как дополнительные ограничения.
- Для stackable-лотов продавца, которые точно совпали со static entry из поддерживаемой poe.ninja категории, можно использовать агрегированную poe.ninja цену как быстрый market estimate; для rare/unique/equipment не подменяй сравнение похожих `trade2` listings агрегатом.
- Любые торговые выводы должны учитывать spread, малый объем, устаревшие/фейковые listings, комиссии/практическую исполнимость и риск price fixing.
- Для прибыльности разделяй данные: raw listings, нормализованные цены, агрегаты/медианы, выводы/сигналы. Не смешивай это в одном неявном словаре.
- Диагностику качества данных и проверку сигналов по истории строй по сохраненным `market_history` снимкам: свежесть, покрытие, рисковые строки и результат прошлых сигналов должны быть отдельным диагностическим payload, а не скрытой частью торгового вывода.
- Для Liquid Emotions считай не только соседний апгрейд, а все пути `3^k x N -> N+k`, чтобы находить лучшую точку входа и выхода в цепочке преобразований. В рекомендациях скрывай продолжение пути, если более короткий выход из той же входной эмоции дает не худшее качество сигнала и большую или равную маржу. Splinter-to-entry рецепты и boss-entry комплекты держи в `app.recipes`: обычные рецепты требуют priced вход и priced выход, а entry-set без продаваемого результата показывай как стоимость входа, не как прибыль.
- Регулярные слепки stackable-рынка и чистых предметных основ (`ItemBases`) собирай через `app.market_snapshots` / `python -m app.cli market-snapshots`: для `ItemBases` по умолчанию используй мгновенный выкуп `status=securable`, основной target `exalted`, 5 минут первые 48 часов при заданном `--league-start`, затем 15 минут. История слепков хранится в SQLite `data/poe2_ninja.sqlite` / `market_history`; `granularity` разделяет `raw`, `hourly` и `daily`, compaction выполняется через `app.history_compaction` / `python -m app.cli compact-history` или фоновый `app.market_service`. Старый `data/trade_rate_history.jsonl` используй только как источник миграции или legacy fallback. Пока FastAPI-приложение активно, `app.market_service` запускает сбор по умолчанию и раз в 10 минут проверяет новые PoE2 trade-лиги. Не трактуй `volume` как точное количество закрытых сделок; это прокси спроса/активности.
- Для `ItemBases` основной список чистых normal-основ бери из PoB PoE2 (`D:\Soft\PathOfBuilding-PoE2\Data\Bases`) через `scripts/build_item_base_catalog_seed_from_pob.py`: генератор пишет tracked seed `app/resources/item_base_catalog_seed.json` и локальный runtime-снимок `data/item_base_catalog.json`, а русские видимые названия добирает из PoE2DB. Не возвращайся к старому index-based сопоставлению `trade2/data/items` EN/RU для списка основ: оно может сдвигать переводы. Из PoB включай только allowlist equipment-файлы: оружие, доспехи/offhand, бижутерия и талисманы; исключай `DNT/UNUSED`, `not_for_sale`, `hidden`, синтетические варианты в скобках, валюту, ключи, инстансные предметы, флаконы/чармы, самоцветы, удочки, ядра душ, trap tools и прочий не-equipment мусор. Общий экран без фильтра должен строиться из сохраненного PoB-каталога и дозаполнять цены порционным грубым проходом через `trade2/search` + первые 10 priced-лотов, сохраняя накопленные цены между проходами и соблюдая общий header-aware limiter; в UI/API выводи только основы с найденными clean priced лотами/ценой или явной ошибкой сбора, а пустые будущие позиции без рыночного подтверждения не показывай. Точный проход по конкретной основе через `trade2/search` и первые 100 priced-лотов запускай только для введенного фильтра/уточнения. Так как `query_type` для PoB-основ английский, поисковые и fetch-запросы рынка основ выполняй через `https://www.pathofexile.com/api/trade2`, а русские видимые названия бери из seed/PoE2DB. Иконки основ держи локально в `storage/icons/item-bases` и отдавай через `/icons/...`: fallback-иконки генерируй как локальные SVG, реальные trade2-иконки кэшируй при появлении fetched listing. Не блокируй UI длинным `Retry-After`: если trade2 просит ждать сотни секунд, возвращай понятную ошибку/частичный результат, продолжай показывать сохраненный каталог основ и дозаполняй цены фоном. Фильтр по имени без live-обновления должен фильтровать сохраненный/кэшированный рынок, а явное live-обновление по введенному названию должно делать точный поиск конкретной базы через `trade2/search`. В выводе рынка основ сохраняй исходную валюту лота и отдельный пересчет в выбранную валюту оценки через сохраненные `Currency`-снимки; показывай нижнюю цену как ориентир низа рынка, оптимум по первой сотне как более практичную цену и не скрывай дешевые лоты только как `<0.01 exalted`. Даже fallback-каталог основ должен иметь русские видимые названия, русские группы и иконки; английские `Fallback`/base names не выводи в русской версии UI.
- Live UI должен читать актуальные сохраненные слепки через `/api/trade/category-rates/latest` и автообновлять открытую категорию при появлении более нового `created_ts` в SQLite. Принудительные live-запросы к poe.ninja/trade2 оставляй за явными действиями пользователя или фонового сборщика, а не за частым браузерным polling.
- Стартовые справочники `/api/trade/leagues` и `/api/trade/static` не должны ломать UI ответом `502`: при временном сбое trade2 отдавай локальный bootstrap fallback без поля `error`, чтобы браузер не проваливался в пустой минимальный режим.
- В UI цены в сферах/валютах округляй до сотых; внутренние расчеты, история и нормализация могут хранить исходную точность, но пользователю не показывай длинные дробные хвосты вроде `0.10064054`.
- Личный кабинет - локальная функция поверх SQLite: пользователи, сессии, закрепленные позиции, Telegram-уведомления, админские права, локальные метрики ИИ и журнал сделок не требуют OAuth PoE. Новая регистрация требует подтверждения email; SMTP настраивается через `.env`, а без SMTP допустима локальная dev-ссылка подтверждения. Telegram bot token хранится в `.env`, пользователь указывает chat id и правила по закрепленным позициям. Уведомления проверяй через `app.notification_worker` после новых market snapshots или `python -m app.cli notifications-check`, чтобы Telegram-события не зависели от открытого UI. Админский bootstrap идет через `ADMIN_USERNAME`/`ADMIN_PASSWORD`; если в базе нет админов, первый существующий или первый зарегистрированный пользователь получает админку. Веб-ИИ ограничивается локальной дневной квотой `AI_DAILY_QUOTA`; это не фактическая квота внешнего провайдера.
- Сделки разделяй на отслеживание, открытый вход и закрытый выход. Номинальная маржа считается в валюте входа. Реальная маржа считается через benchmark: `(current_value / entry_value) / (current_benchmark / entry_benchmark) - 1`. На старте сезона не зашивай Divine Orb как единственный смысловой эталон: UI должен позволять выбрать Divine/Exalted/Chaos и `basket:liquid-core`; составной benchmark держи в `app.benchmark`, не размазывай веса по UI. Сводный P/L-отчет не должен скрыто конвертировать валюты: группируй номинальный и benchmark-adjusted результат, комиссии и стратегии по валютам.
- Рублевый слой FunPay держи скрытым за пользовательским `fiat_rub_enabled` в личном кабинете. Используй только публичную витрину `https://funpay.com/chips/209/`, запрашивая ее с cookie `cy=rub`; не ходи по `/chips/offer`, `/users`, `/chat`, `/trade` и не требуй `golden_key` для рыночной оценки. Сохраняй сырые офферы и агрегаты отдельно от PoE2 `market_history`; изменения `stock/offers` трактуй только как прокси спроса/предложения, не как подтвержденные покупки. Ориентир RUB-курса считай по низу ликвидного рынка, отсекай явно завышенные/старые цены и не превращай профильную вкладку в список офферов, оставляй ссылку на источник для ручной проверки. Автоматический сбор RUB-базы идет тем же циклом, что и `market_snapshots`, через `FUNPAY_RUB_SNAPSHOT_ENABLED` / `FUNPAY_RUB_SNAPSHOT_TARGET`. Рекомендации дня недели и часов покупки/продажи Divine за ₽ выводи только из накопленных почасовых снимков и помечай низкую уверенность, пока истории мало.
- Для ИИ-анализа рынка сначала собирай контекст через `app.ai_context` / `/api/ai/market-context`: передавай снимки, агрегаты, внешние новости и риски как данные, но не позволяй ИИ выдумывать цены или автоматически покупать.
- Валютный анализ держи в `app.currency_analyzer` / `/api/trade/currency-analysis` / `python -m app.cli currency-analyze`: локально считай историю, тренд, волатильность и осторожный прогноз по сохраненным снимкам Currency. ИИ передавай этот контекст через `/api/ai/currency-analysis`, но не позволяй ему заменять расчет цен или прогнозов.
- В live UI вкладка ИИ должна показываться только пользователям с `can_use_ai` или админкой. Из браузера запускай анализ через backend-задачу `/api/ai/market-analysis`, а не прямым вызовом Codex CLI.
- Публичный UI держи на каноническом origin `https://xapct.ru:9038`; `https://xapct.ru` без порта должен перенаправляться на порт, кроме `/api/...`. API на `https://xapct.ru/api/...` должен оставаться доступным с CORS и credentials для `https://xapct.ru:9038`, иначе браузерные account-запросы могут ломаться из-за `Alt-Svc`/HTTP3 на публичном порту.
- Codex CLI-анализатор рынка должен идти через `app.codex_market_analyzer` / `python -m app.cli market-analyze`: запускать `codex exec` только в read-only режиме, сохранять входной payload и ответ в `data/ai_market_analyses`, валидировать `action/confidence` и не превращать сигнал в автопокупку.
- Если добавляешь анализ предметов, сначала сделай parser/normalizer для pasted item text или fetched listing, затем отдельный pricing engine. Текущий parser живет в `app.item_parser`; рыночная оценка pasted item идет через `app.trade2.get_pasted_item_market` и похожие public priced listings. Для pasted rare без official stat id допускай fallback по нормализованному тексту аффиксов, но явно показывай низкую/среднюю уверенность и не называй это точной ценой.
- UI держи тонким: логику выноси в API/service слой `app/`, а шаблоны и статику оставляй только для отображения данных.

## Команды проверки

Запускай после изменений в Python-коде:

```powershell
python -m compileall mcp_server.py app
```

Эта команда рекурсивно проверяет синтаксис всех Python-файлов проекта без запуска приложения и без сетевых запросов.

После изменений в Python-коде также запускай unit-тесты:

```powershell
python -m pytest -q
```

`tests/conftest.py` добавляет корень репозитория в `sys.path`, поэтому запускай `pytest` из корня проекта.

Для проверки публичных PoE2 trade2 справочников можно выполнить:

```powershell
@'
import asyncio
import mcp_server

async def main():
    leagues = await mcp_server.trade_leagues()
    static = await mcp_server.trade_static_data()
    static_ru = await mcp_server.trade_static_data(locale="ru")
    print("leagues", len(leagues.get("result", [])))
    print("static categories", len(static.get("result", [])))
    print("static ru categories", len(static_ru.get("result", [])))

asyncio.run(main())
```

## Работа с файлами

- Не удаляй legacy-файлы только потому, что они плохие. Если нет git-истории или явного запроса, сначала зафиксируй в документации, что они legacy.
- Можно удалять сгенерированные артефакты: `__pycache__`, `*.pyc`, `*.log`, `Answer_API.txt`, сохраненные HTML-снимки.
- При добавлении секретов используй `.env`; при добавлении новых переменных обновляй `.env.example`.
- SMTP-пароли, Telegram bot token, отправители и реальные почтовые настройки не выводи в ответы и не коммить.
- Перед правками проверяй `git status --short --branch`.

## Работа с репозиторием

- Ветки `feat/*` и `fix/*` создавай только с явного согласия пользователя. Без запроса не открывай новые ветки под свои изменения; если работаешь на дефолтной ветке и нужен коммит, сначала спроси, создавать ли ветку.
- Общение, комментарии в коде и сообщения коммитов веди на русском языке.
- Репозиторий мультиагентный: с ним работают разные ИИ-агенты (Claude Code, Codex, OpenCode, Cline и т.п.). Не завязывай инструкции, конфигурацию и структуру проекта на один конкретный агент. Общие правила держи в `AGENTS.md`; специфичные для агента файлы (`CLAUDE.md` и подобные) оставляй минимальными указателями на `AGENTS.md`.
- По завершении работ в ветке, отличной от `main`, предлагай пользователю слить все изменения в `main`.

## Коммуникация

- Отвечай по-русски, если пользователь пишет по-русски.
- Предлагая команду, всегда поясняй, для чего она нужна и что должна сделать.
- Для вопросов "нужен ли API-ключ" отвечай конкретно: для `trade2` обычно нет, для официального `api.pathofexile.com` нужен OAuth и конкретный scope.
- Если API мог измениться, проверяй актуальное состояние live-запросом или по официальной документации, а не опирайся на старые логи.
